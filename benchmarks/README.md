# Skill Benchmarks

Reproducible head-to-head comparisons of how AI coding agents perform on the same prompt with and without Amsterdam-specific skills loaded.

> **Live showcase:** [amsterdam.github.io/amsterdam-agent-skills](https://amsterdam.github.io/amsterdam-agent-skills/) — interactive gallery, prev/next through every variant.

Each benchmark is a directory with:

```
benchmarks/{NNN-slug}/
├── benchmark.yaml          # Machine-readable: prompt, variants matrix, rubric
├── benchmark.md            # Human prose: rationale, findings, narrative
├── prototypes/             # One subdirectory per built variant
│   └── {variant-id}/
│       ├── index.html      # Iframable static output
│       ├── assets/
│       ├── meta.json       # Run metadata (model, tokens, time, score)
│       └── session.md      # Agent session transcript
└── legacy-screenshots/     # Optional: pre-runner manual captures
```

The showcase reads `benchmarks.json` (regenerated by `bun bench manifest`) and surfaces every prototype with an overlay toolbar that cycles through variants of the same benchmark.

## How to run a benchmark

The runner is a Bun CLI that wraps the [GitHub Copilot CLI](https://docs.github.com/en/copilot/github-copilot-in-the-cli). Make sure `copilot` is on your `PATH` and authenticated with `copilot login`.

```bash
cd tools/bench
bun install

# List all benchmarks and their variants
bun run src/cli.ts list

# Build one variant
bun run src/cli.ts run 001 --variant copilot_claude-opus-4-6_amsterdam

# Build the entire matrix declared in benchmark.yaml
bun run src/cli.ts matrix 001

# Score every variant of a benchmark with the LLM judge
bun run src/cli.ts judge 001

# Refresh the showcase manifest (the run/judge commands do this automatically)
bun run src/cli.ts manifest
```

Full CLI reference: [`tools/bench/README.md`](../tools/bench/README.md).

## Adding a new benchmark

1. Create `benchmarks/NNN-your-slug/benchmark.yaml`. Copy the shape from `001-zuidas-landing-page/benchmark.yaml`. Required fields:
   - `id`, `slug`, `name`, `difficulty`, `persona`, `skillsTested`
   - `prompt` — the verbatim text fed to the agent
   - `variants` — at least one. Each declares `id`, `label`, `agent`, `model`, `skills`
   - `rubric` — categorized scoring criteria
   - `ratings` — score-range → rating-label mapping for the showcase badge

2. Write `benchmarks/NNN-your-slug/benchmark.md` — human prose: rationale, what to look for, findings.

3. Run the matrix and confirm prototypes land under `prototypes/`:

   ```bash
   cd tools/bench
   bun run src/cli.ts matrix NNN
   ```

4. Refresh the manifest and rebuild the site:

   ```bash
   bun run src/cli.ts manifest
   cd ../../site && bun run build
   ```

5. Open `site/dist/index.html` in a browser to verify your benchmark appears in the grid and the iframe loads.

## Scoring

Three sources of truth, in order of authority:

1. **Human review on PR** — a maintainer overrides scores in `meta.json` if they disagree with the LLM judge.
2. **LLM judge** (`bench judge`) — calls Copilot with the rubric and the rendered HTML, returns structured JSON scores. Advisory.
3. **Original manual scoring** — frozen in `benchmark.md` for benchmarks that pre-date the runner.

Scores have shape `{ total, max, byCategory, criteria, source, scoredAt }` and are written into the prototype's `meta.json`. The showcase ScoreBadge color-codes by the rating band defined in `benchmark.yaml`.
